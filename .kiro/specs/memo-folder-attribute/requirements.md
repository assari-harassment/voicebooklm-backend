# メモのフォルダー属性機能 - 要件定義

## 概要

メモにフォルダー属性を追加し、AI整形時に自動的にフォルダー分類を行う機能。
ユーザーの既存フォルダー構造を参考にしつつ、AIが適切なフォルダーを提案する。
フォルダーはマスターテーブルで管理し、リネーム・移動機能に対応する。

---

## 機能要件

### FR-1: フォルダーの管理

**概要**: ユーザーごとに1〜3階層のフォルダーを管理できる

**期待する動作**:
- フォルダーは `folders` テーブルで管理される
- 各フォルダーは一意のID、名前、親フォルダーIDを持つ
- メモは `folder_id` でフォルダーを参照する
- フォルダー未設定の場合は `folder_id = null`（未分類）

**階層構造の例**:
```
folders テーブル:
  id=1, name="仕事",           parent_id=null
  id=2, name="プロジェクトA",   parent_id=1      → パス: "仕事/プロジェクトA"
  id=3, name="ミーティング",    parent_id=2      → パス: "仕事/プロジェクトA/ミーティング"
  id=4, name="プライベート",    parent_id=null
```

**制約**:
- 階層数の制限なし（ユーザーが自由に作成可能）
- フォルダー名は空文字不可
- フォルダー名の最大長: 50文字
- 同一親フォルダー内で名前の重複不可

---

### FR-2: AI整形時のフォルダー自動分類

**概要**: AI整形時に、メモ内容に基づいてフォルダーを自動分類する

**制約**:
- AIが生成するフォルダーパスは最大3階層まで
- ユーザーが手動で設定するフォルダーには階層制限なし

**期待する動作**:

#### シナリオ2-1: 既存フォルダーから選択
```
前提:
  - ユーザーAの既存フォルダー: ["仕事", "仕事/プロジェクトA", "プライベート"]
  - 新規メモの内容: "プロジェクトAの進捗会議のメモ"

AI整形を実行すると:
  - AIは既存フォルダー一覧を参考にする
  - AIは "仕事/プロジェクトA" を選択
  - メモの folder_id に該当フォルダーのIDが設定される
```

#### シナリオ2-2: 新規フォルダーの作成
```
前提:
  - ユーザーAの既存フォルダー: ["仕事", "プライベート"]
  - 新規メモの内容: "買い物リスト - 牛乳、卵、パン"

AI整形を実行すると:
  - AIは既存フォルダーに適切なものがないと判断
  - AIは "プライベート/買い物" を提案
  - 「買い物」フォルダーが「プライベート」の子として新規作成される
  - メモの folder_id に新規フォルダーのIDが設定される
```

#### シナリオ2-3: 初回ユーザー（既存フォルダーなし）
```
前提:
  - ユーザーBの既存フォルダー: なし
  - 新規メモの内容: "来週の旅行計画"

AI整形を実行すると:
  - AIは自由にフォルダー名を生成
  - AIは "旅行" を提案
  - 「旅行」フォルダーが新規作成される
  - メモの folder_id に新規フォルダーのIDが設定される
```

#### シナリオ2-4: 分類困難な内容
```
前提:
  - メモの内容が曖昧または短すぎる

AI整形を実行すると:
  - folder_id: null（未分類）
```

---

### FR-3: フォルダー一覧の取得

**概要**: ユーザーのフォルダー一覧を階層構造で取得できる

**期待する動作**:
```
ユーザーAが以下のフォルダーを持つ場合:
  - 仕事 (id=1)
    - プロジェクトA (id=2)
    - プロジェクトB (id=3)
  - プライベート (id=4)

フォルダー一覧を取得すると:
  [
    { id: 1, name: "仕事", parentId: null, path: "仕事" },
    { id: 2, name: "プロジェクトA", parentId: 1, path: "仕事/プロジェクトA" },
    { id: 3, name: "プロジェクトB", parentId: 1, path: "仕事/プロジェクトB" },
    { id: 4, name: "プライベート", parentId: null, path: "プライベート" }
  ]
```

---

### FR-4: フォルダーのリネーム

**概要**: フォルダー名を変更できる

**期待する動作**:
```
前提:
  - フォルダー「仕事」(id=1) が存在
  - 子フォルダー「プロジェクトA」(id=2, parent_id=1) が存在
  - メモがそれぞれのフォルダーに紐付いている

「仕事」を「Work」にリネームすると:
  - folders テーブルの id=1 の name が "Work" に変更される
  - 子フォルダー「プロジェクトA」は変更なし
  - メモの folder_id は変更なし
  - パス表示は自動的に "Work/プロジェクトA" になる
```

**制約**:
- 同一親フォルダー内で既に存在する名前にはリネーム不可

---

### FR-5: フォルダーの移動

**概要**: フォルダーを別の親フォルダーに移動できる

**期待する動作**:
```
前提:
  - フォルダー構造:
    - 仕事 (id=1)
      - プロジェクトA (id=2)
    - プライベート (id=3)

「プロジェクトA」を「プライベート」の下に移動すると:
  - folders テーブルの id=2 の parent_id が 3 に変更される
  - メモの folder_id は変更なし
  - パス表示は "プライベート/プロジェクトA" になる
```

**制約**:
- 自分自身の子孫には移動不可（循環参照防止）
- 移動先に同名フォルダーが存在する場合はエラー

---

### FR-6: フォルダーの削除

**概要**: フォルダーを削除できる

**期待する動作**:
```
前提:
  - フォルダー「買い物」(id=5) が存在
  - メモ3件が folder_id=5 で紐付いている

「買い物」フォルダーを削除すると:
  - folders テーブルから id=5 が削除される
  - メモ3件の folder_id が null（未分類）に変更される
```

**制約**:
- 子フォルダーがある場合は削除不可（先に子を削除または移動が必要）

---

### FR-7: フォルダーによるメモ検索

**概要**: 特定のフォルダー（およびサブフォルダー）のメモを検索できる

**期待する動作**:

#### シナリオ7-1: 完全一致検索
```
フォルダー id=2 (仕事/プロジェクトA) で検索すると:
  - folder_id = 2 のメモのみ取得
  - 子フォルダー (id=5: 仕事/プロジェクトA/ミーティング) のメモは含まれない
```

#### シナリオ7-2: 階層以下の全メモ検索
```
フォルダー id=1 (仕事) 以下で検索すると:
  - folder_id = 1 (仕事)
  - folder_id = 2 (仕事/プロジェクトA)
  - folder_id = 5 (仕事/プロジェクトA/ミーティング)
  - 上記すべてのフォルダーに属するメモが取得される
```

#### シナリオ7-3: 未分類メモの検索
```
未分類メモを検索すると:
  - folder_id が null のメモのみ取得
```

---

## 非機能要件

### NFR-1: パフォーマンス
- フォルダー一覧の取得: 100フォルダーまで100ms以内
- フォルダー検索: 適切なインデックスで高速化
- 階層以下の検索: 再帰クエリまたは事前計算で対応

### NFR-2: 後方互換性
- 既存メモの `folder_id` は `null`（未分類）として扱う
- 既存APIのレスポンスにフォルダー情報を追加

### NFR-3: データ整合性
- フォルダー削除時にメモの folder_id を null に更新
- 循環参照を防止

## 影響範囲

### データベース
- `folders` テーブル新規作成
- `memos` テーブルに `folder_id` カラム追加
- 外部キー制約とインデックス追加

### ドメインモデル
- `Folder` エンティティ新規作成
- `FolderRepository` インターフェース新規作成
- `Formatting` 値オブジェクトに `folderId` プロパティ追加
- `VoiceMemo` の関連メソッド更新

### ゲートウェイ
- `MemoFormatResult` に `folderPath` プロパティ追加（AIはパス形式で返す）
- `MemoFormatCommand` に既存フォルダー一覧を追加

### インフラストラクチャ
- `FolderEntity`, `FolderRepositoryImpl` 新規作成
- Gemini AI プロンプトの更新
- `MemoEntity` に `folderId` 追加
- フォルダーパス → ID 変換ロジック

### API
- メモ取得レスポンスに `folder` 情報追加
- フォルダーCRUDエンドポイント新規作成

---

## 設計判断

### 採用する方式
- **マスターテーブル方式**: `folders` テーブルで階層構造を管理
- **外部キー参照**: メモは `folder_id` でフォルダーを参照
- **AIはパス形式で返す**: AIの出力をパス形式 → フォルダーID に変換

### メリット
- フォルダーのリネーム・移動が1レコードの更新で完了
- フォルダーにメタデータ（色、アイコン等）を将来追加可能
- データの整合性を外部キー制約で担保

### トレードオフ
- テーブル追加による複雑化
- AI整形時にフォルダー作成の追加処理が必要
