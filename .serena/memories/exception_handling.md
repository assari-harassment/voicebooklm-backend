# 例外処理アーキテクチャ

## 概要

本プロジェクトでは、オニオンアーキテクチャに基づいた例外処理の仕組みを採用している。
すべての例外は最終的に **プレゼンテーション層** の `GlobalExceptionHandler` で捕捉され、
統一されたエラーレスポンス形式でクライアントに返される。

---

## 例外の分類

### 1. ドメイン例外 (`DomainException`)

**発生箇所**: ユースケース層、ドメイン層

**用途**: ビジネスロジック上のエラーを表現する

**構成要素**:
- `ErrorCode` enum で定義されたエラーコード
- 各エラーコードは HTTPステータス と デフォルトメッセージ を持つ

**現在定義されているエラーコード**:
| コード | HTTPステータス | 説明 |
|--------|---------------|------|
| INVALID_ID_TOKEN | 401 | IDトークン検証失敗 |
| INVALID_REFRESH_TOKEN | 401 | リフレッシュトークン無効/期限切れ |
| USER_NOT_FOUND | 404 | ユーザー未発見 |
| TRANSCRIPTION_FAILED | 422 | 文字起こし処理失敗 |

### 2. リクエスト関連例外

**発生箇所**: プレゼンテーション層（コントローラー）、Spring Framework

| 例外クラス | HTTPステータス | 説明 |
|-----------|---------------|------|
| ResponseStatusException | 可変 | コントローラーで明示的にスロー |
| MissingRequestHeaderException | 401 | 必須ヘッダー（Authorization等）の欠如 |
| MethodArgumentNotValidException | 400 | リクエストボディのバリデーションエラー |

### 3. 予期しない例外

**発生箇所**: すべての層

`Exception` クラスの汎用ハンドラがフォールバックとして捕捉し、500エラーを返す。

## 例外の流れ

```
┌─────────────────────────────────────────────────────────────────┐
│                      クライアント                                │
└─────────────────────────────────────────────────────────────────┘
                              ▲
                              │ JSON レスポンス
                              │ { error, code }
┌─────────────────────────────────────────────────────────────────┐
│              GlobalExceptionHandler (プレゼンテーション層)        │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │ @ExceptionHandler で各種例外をキャッチ                   │    │
│  │ → DomainException      → ErrorCode から HTTP ステータス │    │
│  │ → ResponseStatusException → 指定されたステータスを使用  │    │
│  │ → Validation 例外       → 400                          │    │
│  │ → 予期しない例外        → 500                          │    │
│  └─────────────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────────┘
                              ▲
                              │ 例外がスロー
┌─────────────────────────────────────────────────────────────────┐
│                   プレゼンテーション層                           │
│  (Controller)                                                   │
│  ・入力検証で ResponseStatusException をスロー                   │
│  ・ユースケースを呼び出し                                        │
└─────────────────────────────────────────────────────────────────┘
                              ▲
                              │ DomainException
┌─────────────────────────────────────────────────────────────────┐
│                      ユースケース層                              │
│  (UseCase)                                                      │
│  ・ビジネスルール違反時に DomainException をスロー               │
│  ・インフラ層からの例外を DomainException にラップ               │
└─────────────────────────────────────────────────────────────────┘
                              ▲
                              │ インフラ固有の例外
┌─────────────────────────────────────────────────────────────────┐
│                      インフラ層                                  │
│  (Repository実装, 外部APIクライアント)                           │
│  ・外部サービス呼び出しで発生した例外をログ出力                   │
│  ・必要に応じて呼び出し元に例外を伝播                             │
└─────────────────────────────────────────────────────────────────┘
```

---

## 各層での例外処理ルール

### インフラ層

- 外部サービス（DB、外部API）の例外は **そのまま上位に伝播** させる
- 必要に応じてログを出力する
- 例外の握りつぶし（catch して何もしない）は禁止

### ユースケース層

- ビジネスロジックのエラーは `DomainException` をスローする
- インフラ層からの例外は、必要に応じて `DomainException` にラップする
  - 例: 文字起こしAPIの失敗 → `TRANSCRIPTION_FAILED`
- **原因となる例外は `cause` パラメータで保持** することでスタックトレースを維持

### プレゼンテーション層（コントローラー）

- 入力検証の失敗は `ResponseStatusException` でスローしてよい
- ユースケースから発生した例外は **そのまま上位（GlobalExceptionHandler）に伝播** させる
- コントローラー内での try-catch は必要最小限にする

### GlobalExceptionHandler

- すべての例外を捕捉し、統一されたレスポンス形式に変換
- ログレベルの使い分け:
  - ビジネスエラー → `warn`
  - 予期しないエラー → `error`（スタックトレース付き）

---

## エラーレスポンス形式

```json
{
  "error": "エラーメッセージ",
  "code": "ERROR_CODE"
}
```

---

## 新しいエラーを追加する手順

1. `ErrorCode` enum に新しいエラーコードを追加（HTTPステータスとデフォルトメッセージを指定）
2. 該当のユースケースで `throw DomainException(ErrorCode.NEW_ERROR_CODE)` を呼び出す

これだけで `GlobalExceptionHandler` が自動的に適切なHTTPレスポンスを返す。

---

## 注意事項

- **例外の握りつぶし禁止**: catch ブロックで例外を無視しないこと
- **適切な粒度**: 1つのエラーコードは1つのビジネス上の意味を持つように設計
- **ログの一貫性**: すべての例外は GlobalExceptionHandler でログ出力されるため、各層での重複ログは避ける
