# アーキテクチャ設計思想

## 概要

本プロジェクトは**オニオンアーキテクチャ**と**ドメイン駆動設計（DDD）**の考え方を部分的に採用している。
完全な DDD の導入ではなく、実用性を重視しつつ以下の恩恵を受けることを目的としている：

- ビジネスロジックと技術的関心事の分離
- テスタビリティの向上
- 変更に強い依存関係の構築

---

## 依存関係のルール

依存は常に**外側から内側へ**向かう。内側の層は外側の層を知らない。

```
┌────────────────────────────────────────────────────────────┐
│                   Presentation（最外層）                    │
│  ・HTTPリクエストの受付と応答                               │
│  ・入力のバリデーション                                     │
│  ・エラーハンドリング                                       │
├────────────────────────────────────────────────────────────┤
│                   Infrastructure                           │
│  ・外部サービスとの接続                                     │
│  ・データベースアクセスの実装                               │
│  ・ドメイン層で定義されたインターフェースを実装             │
├────────────────────────────────────────────────────────────┤
│                      UseCase                               │
│  ・アプリケーション固有のビジネスフローを実行               │
│  ・トランザクション境界の管理                               │
│  ・ドメインモデルとリポジトリ/ゲートウェイの協調            │
├────────────────────────────────────────────────────────────┤
│                    Domain（最内層）                         │
│  ・ビジネスルールの表現                                     │
│  ・ドメインモデル、リポジトリ/ゲートウェイのインターフェース │
│  ・外部への依存を持たない                                   │
└────────────────────────────────────────────────────────────┘
```

---

## 各層の役割と制約

### Domain 層

**役割**: ビジネスの核心となる概念とルールを表現する

**含まれるもの**:
- ドメインモデル（エンティティ、値オブジェクト）
- リポジトリインターフェース（永続化の抽象）
- ゲートウェイインターフェース（外部サービス連携の抽象）
- ドメイン例外

**制約**:
- フレームワークや外部ライブラリに依存しない
- 他のどの層も参照しない（最も内側）
- 永続化方法や外部サービスの詳細を知らない

**設計指針**:
- ドメインモデルは不変オブジェクトとして設計し、状態変更は新しいインスタンスを返す
- ビジネスルールの検証はモデル内で行い、不正な状態を許容しない
- リポジトリとゲートウェイはインターフェースのみ定義し、実装を持たない

---

### UseCase 層

**役割**: アプリケーション固有のビジネスフローを実行する

**含まれるもの**:
- ユースケースクラス（1クラス1責務）
- 入出力用のデータ構造

**制約**:
- ドメイン層にのみ依存する
- HTTPやデータベースなどの技術的詳細を知らない
- プレゼンテーション層の都合（レスポンス形式など）に引きずられない

**設計指針**:
- ユースケースは「何をするか」を表す名前を持つ
- 複数のドメインモデルやリポジトリを協調させてフローを完成させる
- トランザクションの開始と終了はこの層で管理する
- ID生成などのインフラ寄りの処理もこの層で行い、ドメインモデルには渡すだけにする

---

### Infrastructure 層

**役割**: 外部世界との接続を担う

**含まれるもの**:
- リポジトリの実装（データベースアクセス）
- ゲートウェイの実装（外部API呼び出し）
- 認証・認可の実装
- フレームワーク固有の設定

**制約**:
- ドメイン層で定義されたインターフェースを実装する
- ドメインモデルへの変換責務を持つ（DBエンティティ ↔ ドメインモデル）
- ユースケース層やプレゼンテーション層を参照しない

**設計指針**:
- 外部サービスの障害やエラーはログを出力し、適切に上位へ伝播させる
- 永続化用のエンティティとドメインモデルは明確に分離する
- フレームワーク固有のアノテーションはこの層に閉じ込める

---

### Presentation 層

**役割**: 外部からのリクエストを受け付け、レスポンスを返す

**含まれるもの**:
- コントローラー
- リクエスト/レスポンス用のデータ構造
- グローバルエラーハンドラー

**制約**:
- ユースケース層を呼び出す
- ドメインモデルを直接返さず、レスポンス用の構造に変換する
- ビジネスロジックを持たない

**設計指針**:
- 入力検証（形式チェック）はこの層で行う
- ビジネスルール違反の判定はユースケース/ドメイン層に委ねる
- 例外はグローバルハンドラーで統一的に処理する

---

## 採用しているDDDの概念

| 概念 | 適用状況 |
|------|----------|
| 集約（Aggregate） | VoiceMemoを集約ルートとして設計。関連するTranscription/Formattingを内包 |
| リポジトリ | 集約単位での永続化インターフェースをドメイン層に定義 |
| 不変オブジェクト | ドメインモデルは状態変更時に新インスタンスを返す |
| ゲートウェイ | 外部サービス連携の抽象化（ポート&アダプタに相当） |
| ドメイン例外 | ビジネスルール違反を表現する専用の例外クラス |

---

## 意図的に採用していないもの

- **境界づけられたコンテキスト**: 単一のマイクロサービスとして構築しているため明示的な分離は行わない
- **ドメインサービス**: ユースケースがその役割を担う
- **ドメインイベント**: 現時点では非同期処理の要件がないため未導入
- **仕様パターン**: 複雑なクエリ条件が必要になった時点で検討

---

## 補足: なぜ「部分的」なのか

フルスペックのDDDは学習コストと実装コストが高い。
本プロジェクトでは以下のバランスを取っている：

- **採用する部分**: 依存関係の方向制御、ドメインモデルの純粋性、リポジトリ/ゲートウェイによる抽象化
- **簡略化する部分**: 境界づけられたコンテキストの分離、ドメインイベントによる結合度低減

規模が拡大した際には、必要に応じて追加の概念を導入する余地を残している。
